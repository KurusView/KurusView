Using GoogleTest from various build systems

GoogleTest comes with pkg-config files that can be used to determine all necessary flags for compiling and linking to GoogleTest (and GoogleMock). Pkg-config is a standardised plain-text format containing

  • the includedir (-I) path
  • necessary macro (-D) definitions
  • further required flags (-pthread)
  • the library (-L) path
  • the library (-l) to link to

All current build systems support pkg-config in one way or another. For all examples here we assume you want to compile the sample samples/sample3_unittest.cc.

CMake

Using pkg-config in CMake is fairly easy:

1 cmake_minimum_required(VERSION 3.0)
2 
3 cmake_policy(SET CMP0048 NEW)
4 project(my_gtest_pkgconfig VERSION 0.0.1 LANGUAGES CXX)
5 
6 find_package(PkgConfig)
7 pkg_search_module(GTEST REQUIRED gtest_main)
8 
9 add_executable(testapp samples/sample3_unittest.cc)
10 target_link_libraries(testapp ${GTEST_LDFLAGS})
11 target_compile_options(testapp PUBLIC ${GTEST_CFLAGS})
12 
13 include(CTest)
14 add_test(first_and_only_test testapp)

It is generally recommended that you use target_compile_options + _CFLAGS over target_include_directories + _INCLUDE_DIRS as the former includes not just -I flags (GoogleTest might require a macro indicating to internal headers that all libraries have been compiled with threading enabled. In addition, GoogleTest might also require -pthread in the compiling step, and as such splitting the pkg-config Cflags variable into include dirs and macros for target_compile_definitions() might still miss this). The same recommendation goes for using _LDFLAGS over the more commonplace _LIBRARIES, which happens to discard -L flags and -pthread.

Autotools

Finding GoogleTest in Autoconf and using it from Automake is also fairly easy:

In your configure.ac:

1 AC_PREREQ([2.69])
2 AC_INIT([my_gtest_pkgconfig], [0.0.1])
3 AC_CONFIG_SRCDIR([samples/sample3_unittest.cc])
4 AC_PROG_CXX
5 
6 PKG_CHECK_MODULES([GTEST], [gtest_main])
7 
8 AM_INIT_AUTOMAKE([foreign subdir-objects])
9 AC_CONFIG_FILES([Makefile])
10 AC_OUTPUT

and in your Makefile.am:

1 check_PROGRAMS = testapp
2 TESTS = $(check_PROGRAMS)
3 
4 testapp_SOURCES = samples/sample3_unittest.cc
5 testapp_CXXFLAGS = $(GTEST_CFLAGS)
6 testapp_LDADD = $(GTEST_LIBS)

Meson

Meson natively uses pkgconfig to query dependencies:

1 project('my_gtest_pkgconfig', 'cpp', version : '0.0.1')
2 
3 gtest_dep = dependency('gtest_main')
4 
5 testapp = executable(
6  'testapp',
7  files(['samples/sample3_unittest.cc']),
8  dependencies : gtest_dep,
9  install : false)
10 
11 test('first_and_only_test', testapp)

Plain Makefiles

Since pkg-config is a small Unix command-line utility, it can be used in handwritten Makefiles too:

``makefile GTEST_CFLAGS =pkg-config –cflags gtest_main GTEST_LIBS =pkg-config –libs gtest_main`

.PHONY: tests all

tests: all ./testapp

all: testapp

testapp: testapp.o $< -o $@

testapp.o: samples/sample3_unittest.cc $< -c -o $@

2 
3 Let's say you have a `CMakeLists.txt` along the lines of the one in this
4 tutorial and you try to run `cmake`. It is very possible that you get a failure
5 along the lines of:

– Checking for one of the modules 'gtest_main' CMake Error at /usr/share/cmake/Modules/FindPkgConfig.cmake:640 (message): None of the required 'gtest_main' found

1 These failures are common if you installed GoogleTest yourself and have not
2 sourced it from a distro or other package manager. If so, you need to tell
3 pkg-config where it can find the `.pc` files containing the information. Say you
4 installed GoogleTest to `/usr/local`, then it might be that the `.pc` files are
5 installed under `/usr/local/lib64/pkgconfig`. If you set

export PKG_CONFIG_PATH=/usr/local/lib64/pkgconfig ```

pkg-config will also try to look in PKG_CONFIG_PATH to find gtest_main.pc.
